API Specification / Security Notes

OpenAPIは「点」を描き、
Arazzoは「線」を描く。

単一エンドポイントの仕様であるOpenAPIと、複数エンドポイントをまたぐ呼び出し順序を定義するArazzo。この記述単位の違いが、なぜ「ロジックの脆弱性」の発見において決定的な意味を持つのかを整理する。

1. まず、両者は何を記述するものか overview

OpenAPI Specification（OAS）は、個々のHTTPエンドポイントの入出力を記述する。パス、メソッド、パラメータ、リクエスト/レスポンスのスキーマ、認証方式などを定義するが、「どのエンドポイントを、どの順番で、どんな条件で呼ぶべきか」は表現できない。

Arazzo Specification（OpenAPI Initiativeが策定）は、その上位レイヤーとして、複数のAPI呼び出しを一つの「ワークフロー」として結び付ける。あるステップの出力を次のステップの入力にマッピングする方法、各ステップの成功/失敗の判定条件、条件分岐やリトライなどを宣言的に記述する。

2. OpenAPIだけでは見えないもの gap

OpenAPIの仕様書には、たとえば「注文を確定する前にカートへ商品を追加し、クーポンを適用する必要がある」という業務上の前提条件は書かれていない。各エンドポイントは独立した項目として並んでいるだけで、それらの間にある因果関係・順序制約・状態遷移は仕様書を読む人間の頭の中にしか存在しない。

この「仕様に書かれない前提」こそが、ロジックの脆弱性（ビジネスロジック脆弱性）の温床になる。IDOR、認可の抜け穴、レースコンディション、決済フローの迂回、状態遷移の不整合などは、単体のエンドポイントの入出力検証（型・スキーマ）を通過しても発生しうる。単体テストや静的解析が強いのはスキーマレベルの誤りであり、「本来この順番で呼ばれるべきAPIを、順番を変えて呼んだらどうなるか」という問いには答えてくれない。

3. Arazzoがロジック脆弱性の発見に寄与する仕組み how

Arazzoは脆弱性スキャナーそのものではないが、「意図された正規のフロー」を機械可読な形で固定することで、そこからの逸脱を検出可能にする土台を提供する。具体的には次の4点が効いてくる。

01

正規順序が形式化される

ステップの実行順序と依存関係が明文化されるため、「途中のステップを飛ばして呼ぶ」「順序を入れ替えて呼ぶ」といったテストケースを、仕様から機械的に総当たり生成できる。人間が想像力で見つけていた迂回経路を、網羅的に洗い出せる。

02

入出力マッピングが可視化される

あるステップの出力（例：注文ID、割引率）が次のステップの入力にどう渡されるかが明示される。この受け渡し箇所こそ、値の改ざんや検証漏れ（例：クライアントが送る割引率をサーバが再検証しない）が起きやすい境界であり、Arazzoはその境界そのものをリスト化してくれる。

03

成功/失敗条件が宣言的になる

各ステップのsuccessCriteriaが明示されるため、「本来失敗すべき遷移が、実際には200 OKを返してしまう」というギャップを、仕様と実挙動の差分として機械的に検出できる。期待値がコードやドキュメントの奥ではなく、仕様の一部として存在することが重要。

04

AIエージェント/自動テストの入力になる

Arazzo文書はAIエージェントやテスト生成ツールがワークフローを機械的に解釈するための入力になる。これにより「正規フロー通りのリクエスト」と「そこから外れた攻撃的なリクエスト」の両方を、人手を介さず大量に生成・実行できる。

4. 具体例で見る：迂回パスの発見 example

「カート追加 → クーポン適用 → 支払い」という3ステップの購入フローをArazzoで定義したとする。正規のフローは以下の通り。

POST /cart/items
→
POST /cart/coupon
→
POST /checkout
Arazzoが定義する正規順序
POST /checkout
先頭ステップをスキップして直接呼ぶ
Arazzoが依存関係を「checkoutはcart.itemsとcart.couponの出力を入力とする」と定義していれば、この依存を満たさない直接呼び出しを異常系として自動生成・検証できる。もし checkout エンドポイント単体が前提条件をチェックしていなければ、カートが空のまま決済が通ってしまう、あるいは割引の二重適用が起きるといったロジック脆弱性が、ここで顕在化する。

Arazzoの記述イメージ（簡略化）：

# checkout ステップが前段の出力に依存していることの宣言
- stepId: checkout
  operationId: createOrder
  dependsOn: [addItem, applyCoupon]
  successCriteria:
    - condition: $statusCode == 200 && $response.body.total > 0

このように「依存関係」と「成功条件」が仕様として存在することで、依存を無視したリクエストや、成功条件を満たさないのに200が返るケースを、診断ツールが機械的に検出できるようになる。

5. 過信してはいけない点 limits

Arazzoは「脆弱性を見つけるツール」ではなく「見つけやすくする仕様」にすぎない。Arazzo文書があるだけでは何も検証されず、それを解釈してテストケースを生成・実行する診断ツールやテストフレームワーク側の実装があって初めて効果を発揮する。

また、正規フローが明文化されることは諸刃の剣でもある。防御側がテスト生成に使えるのと同様に、攻撃者にとっても「このAPI群はどう組み合わさっているか」を推測する手間が減る、という側面は認識しておく必要がある。加えて、Arazzoが表現できるのはあくまで設計者が想定した依存関係であり、設計者自身が見落としているロジック上の抜け穴（想定していない状態遷移そのもの）までは、仕様に書かれない限り自動では見つからない。
