SPEC INSPECTION LOG · OpenAPI

API定義書は、
何を検査されているのか

ガバナンスルールが自動でチェックしている3つの観点を、OpenAPIの記述例とともに読み解く。

OpenAPIのようなAPI定義書は、自動化されたガバナンスチェックの対象になる。多くの人は「インターフェース契約」——パスやメソッド、ステータスコードといった実際の仕様——だけを検査対象だと考えがちだが、検査できる情報には実はもう2種類ある。この3種類をバランスよく見ることが、価値を生むAPIづくりの土台になる。

原典: Arnaud Lauret, Postman Blog (2023.08.07) 要約・再構成: Claude
API GOVERNANCE3 CHECKSVERIFIED
CHECK 01
インターフェース契約 — APIコール中に実際に交換される要素（パス・メソッド・ステータスコード・データモデル）
CHECK 02
メタデータ — 契約そのものではないが、意味や使い方を伝える情報（説明文・例・operationIdなど）
CHECK 03
仕様の使われ方 — 同じ契約でも記述の仕方次第で変わる、保守性やツール互換性の問題
01
Interface Contract

実際のインターフェース契約を検査する

APIコールの中で実際にやり取りされる要素——リソースパス、HTTPメソッド、ステータスコード、データモデル——が対象。ここを検査することで得られる効果は多岐にわたる。

相互運用性
HTTPの標準的な作法から外れていないかを見る。たとえばGETリクエストに本文が付いていないか、204 No Contentがボディを返していないかを検査する。
使いやすさ
プロパティ名の大文字小文字の揺れや、必須クエリパラメータの妥当性を見る。someName1 / someName2 のような番号付き命名は配列で表現できないかを問い直す。
セキュリティ
すべての操作にセキュリティ定義とスコープが紐づいているか、APIキーのような弱い認証方式に頼りすぎていないかを確認する。
性能
検索系エンドポイントにページネーションや絞り込みパラメータがあるか、標準的なHTTPキャッシュの仕組みが定義されているかを見る。
破壊的変更の予防
配列やプリミティブ型を直接返すのではなく、拡張しやすいオブジェクト構造を使っているかを確認する。
監視のしやすさ
想定外のステータスコードは不要なアラートを誘発するため、使用してよいステータスコードを限定する。

用語解説

204 No Content
処理は成功したがレスポンス本文を返さないことを示すHTTPステータスコード。ボディがあると仕様違反になる。
security scheme / scope
OpenAPIで認証方式（APIキー・OAuth2など）と、その中でどこまでの権限を許可するかを宣言する仕組み。
openapi.yaml — 契約レベルのルールに沿った例YAML
paths:
  /orders:
    get:
      summary: 注文一覧を取得
      security:
        - bearerAuth: []          # すべての操作にセキュリティ定義を必須化
      parameters:
        - name: limit
          in: query
          schema: { type: integer, maximum: 100 }
        - name: cursor
          in: query
          schema: { type: string } # 検索系には必ずページネーション
      responses:
        '200':
          description: "注文一覧。件数が多い場合はcursorで続きを取得できる"
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/OrderList'
        '204':
          description: "該当する注文が存在しない"
                                    # 204は本文なし、を linter がチェック
02
Metadata

APIのメタデータを検査する

設計の良いAPIは契約だけで自明なこともあるが、定義書にはAPIの背景や使い方を伝える情報も含まれる。ドキュメント用の説明文が代表格だが、コード生成やゲートウェイ設定に使われるoperationIdのような設定情報もここに含まれる。

誤ったAPI設計の防止
API名が単に「API」になっていないかを問う。目的を明確にする名前を求めることで、後から本来のスコープ外の機能が付け足される事態を防げる。担当者の連絡先があるかも重要。
理解のしやすさ
操作のsummaryが付いているか、タグで整理されているか、レスポンスの説明が単なるHTTPステータス名の繰り返しになっていないかを確認する。
価値提供までの速さ
リクエスト・レスポンスの例（example）が用意されているかを見る。例があればモックサーバーが作れ、設計段階のフィードバックや利用者の開発を早められる。
扱いやすいコード生成
operationIdが重複なく設定されているかを確認する。ここが揃っていないと、コード生成ツールが読みにくいメソッド名を作ってしまう。

用語解説

operationId
各操作に付ける一意な識別子。SDKやクライアントコードを自動生成する際のメソッド名の元になる。
example
リクエストやレスポンスの具体的な値の見本。モックサーバー生成やAPIドキュメントの分かりやすさに直結する。
openapi.yaml — メタデータレベルのルールに沿った例YAML
info:
  title: "注文管理API"       # 「API」のような無意味な名前をlinterが拒否
  description: "店舗の注文作成・照会・キャンセルを扱う"
  contact:
    name: "Order Platform Team"
    email: "order-platform@example.com"

paths:
  /orders/{orderId}:
    get:
      operationId: getOrderById   # 一意なoperationIdを必須化
      summary: "注文IDで1件取得"
      responses:
        '200':
          description: "指定した注文の詳細"
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Order'
              examples:
                sample:
                  value: { id: "ord_042", status: "shipped" }
03
Specification Usage

仕様の「使われ方」を検査する

同じインターフェース契約でも、OpenAPIのようなフォーマットの中での書き方は何通りもある。その書き方次第で、定義書の一貫性・保守性・使いやすさが変わってくる。

保守性と使いやすさ
リクエスト・レスポンスの構造を毎回書き下ろすのではなく、再利用可能なデータモデル（コンポーネント）として定義しているか、4XX・5XX全体で標準のエラーモデルを使い回しているかを確認する。
ツール互換性
利用しているOpenAPIの機能が、社内で使っているツール群でサポートされていない機能でないかを確認する。

用語解説

$ref
OpenAPIの中で、他の場所に定義したスキーマを参照する記法。同じ構造を何度も書かずに再利用できる。
components/schemas
再利用するデータモデルをまとめて定義しておく場所。$refから参照される。
openapi.yaml — 仕様の使い方レベルのルールに沿った例YAML
components:
  schemas:
    Order:
      type: object
      properties:
        id: { type: string }
        status: { type: string }
    Error:                         # 全エラーで使い回す共通モデル
      type: object
      properties:
        code: { type: string }
        message: { type: string }

paths:
  /orders/{orderId}:
    delete:
      responses:
        '404':
          description: "注文が見つからない"
          content:
            application/json:
              schema: { $ref: '#/components/schemas/Error' }
        '409':
          description: "すでに発送済みでキャンセル不可"
          content:
            application/json:
              schema: { $ref: '#/components/schemas/Error' }
                                    # インラインで書き散らさず、同じErrorを再利用

一つの観点だけを見てはいけない

インターフェース契約はAPI定義書という氷山の中でいちばん目に見える部分にすぎない。そこだけを検査していると、価値創出を支えるはずの重要な側面——メタデータや仕様の使い方——を見落としてしまう。自動化されたガバナンスチェックと定義書のリンティングは、APIライフサイクル全体にわたって、高品質なAPIの作成・保守・利用のプロセスを支える基盤になる。
